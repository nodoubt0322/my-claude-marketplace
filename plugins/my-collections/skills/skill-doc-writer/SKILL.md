此 skill 為已安裝的 Claude Code skill 生成完整的繁體中文使用文檔，讓用戶快速掌握 skill 的功能與最佳使用方式。

## 撰寫哲學

一份好的 skill 說明文件，必須同時做到三件事：

1. **問題優先 (Problem-first)** — 讀者打開文件的第一秒，就要知道「這個 skill 是要解決我的什麼痛」。功能清單放在最後，不是最前。
2. **真實 prompt (Real prompts)** — 範例必須是讀者可以直接複製貼到對話框使用的完整 prompt，不是只列指令名稱。
3. **劃界比堆功能重要 (Boundaries over features)** — 明確說「不適合什麼時候用」「容易與哪個 skill 搞混」，比再多列一個功能更有價值。

**禁忌寫法**：
- 開場放行銷話術（「速度快 810 倍」這類）→ 損失專業度
- 範例只重複指令名稱（` ```/cso``` ` 是無效範例）→ 讀者無法判斷怎麼下指令
- 把整個領域教學塞進指南（一個 `/animate` 指令配 50 行 CSS 教學）→ 變成又臭又長
- 用同一個模板硬套到所有 skill → 簡單 skill 應該短，複雜 skill 才需要長

---

## 執行流程

### 步驟 1：確認目標 skill

如果用戶已指定 skill 名稱，直接進入步驟 2。

如果未指定，搜尋以下路徑列出已安裝的 skills 讓用戶選擇：
- 官方 plugins: `~/.claude/plugins/cache/claude-plugins-official/*/*/skills/*/SKILL.md`
- 本機自訂: `~/.claude/skills/*/SKILL.md`

### 步驟 2：讀取 skill 內容

**必讀：** 目標 skill 的 `SKILL.md`

**選讀（若存在）：**
- `references/` 目錄中的文件 — 了解 skill 依賴的參考資料
- `scripts/` 目錄 — 了解有哪些工具腳本可用
- `assets/` 目錄 — 了解有哪些範本或資源

### 步驟 3：判斷 skill 規模並選擇模板

依 skill 複雜度套用不同模板，**不要硬套**：

| Skill 類型 | 範例 | 模板選擇 |
|---|---|---|
| 單一功能型 | `caveman`、`handoff` | **簡版**：功能概述 + 啟動 + 1-2 範例 + 注意事項 |
| 多階段流程型 | `diagnose`、`tdd` | **標準版**：完整五段式 + 流程拆解 |
| 多子指令型 | `impeccable`、`gstack` | **指令手冊版**：總覽表 + 分類 + 各指令展開 + 工作流組合 |

### 步驟 4：深入理解 skill（撰寫前的內部問答）

這些問題不需要輸出給用戶，但必須能答出來才動筆：

- 這個 skill 解決什麼**具體痛點**？（不是「提升效率」這種抽象答案）
- 沒有它的話，用戶必須**自己做什麼**？
- 最容易和**哪個其他 skill** 搞混？差異在哪？
- 用戶最容易**搞錯**的操作細節是什麼？
- skill 的輸出物**具體長什麼樣**？（文字？檔案？互動？）

### 步驟 5：生成文檔

按下方對應模板撰寫。完成後：
1. **直接在對話中顯示完整文檔**
2. **儲存到** `./AI/skills_guide/<skill-name>-doc.md`（目錄不存在則建立）
3. 告知用戶儲存路徑

## 撰寫高品質內容的具體規則

### 規則 1：範例 prompt 必須是「可直接複製貼上」

❌ 差的範例：
```
/impeccable craft a page
```

✅ 好的範例：
```
/impeccable craft 給獨立音樂人用的個人網站首頁，呈現 90 年代 cassette tape 復古質感，
需要音樂作品展示區、演出場次列表、聯絡表單。用純 HTML/CSS，不要框架。
```

### 規則 2：「解決的是 X 問題」必須是真實痛點原話

❌ 抽象：「提升開發效率」「改善程式碼品質」
✅ 具體：「看了半天程式碼卻找不到 bug 在哪」「計畫聽起來很好，但和程式碼實際結構脫節」

### 規則 3：anti-pattern 比正向描述有力

如果原 SKILL.md 有「NEVER」「絕對禁止」「不要這樣做」等內容，**一定要保留**在文檔裡。這是區分專業指南和維基百科的關鍵。

範例（從 impeccable 提煉）：
> **`bolder` 不是**加 cyan/purple gradient 或 glassmorphism，那是 AI slop 的「bold」。真正的 bolder 是：極端的 font size 對比、900 weight 配 200 weight、讓單一大膽顏色佔 60% 版面。

### 規則 4：避免變成領域教學

如果某個指令的說明需要塞 30 行 CSS code，那就抽出去：
- 改用一句話描述 + 「詳見 [參考連結]」
- 或保留 1-2 行最精華的程式碼示意

文件是**指南**，不是**教程**。

### 規則 5：相似 skill 用比較表

當有兩個 skill 容易搞混（如 `grill-me` vs `grill-with-docs`），務必用表格列出差異：

| | A | B |
|---|---|---|
| 重點 | ... | ... |
| 適用時機 | ... | ... |

### 規則 6：模板長度配合 skill 複雜度

- 1 個觸發指令的 skill → 50-100 行文檔
- 5-10 階段的流程型 skill → 100-200 行
- 30+ 子指令的工具集 → 300-500 行

**寫到一半發現文件比 SKILL.md 還長 3 倍** → 一定是塞太多領域教學進去了，砍。

## 注意事項的來源

從 `SKILL.md` 中尋找以下內容，轉化為文檔的「注意事項」段：

| SKILL.md 中的內容 | 轉化方式 |
|---|---|
| NEVER/ALWAYS/MUST 等強制規則 | 變成「此 skill 不會做 X」或「使用前需要 Y」 |
| 需要特定工具或套件 | 變成「前置需求」 |
| 列出不適用的情境 | 變成「不適合的時機」 |
| 容易與其他 skill 混淆的地方 | 變成比較表 |
| anti-pattern / 反例 | **保留原文**，這是文檔的精華 |
